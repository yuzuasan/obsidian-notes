## フェーズ4：React × TypeScript（実務必須）

### 1. props の型定義

- **props = 親コンポーネントから渡される値**
    
- コンポーネントの「入力の形」を決める
    
- `React.FC` は使わず、普通の関数でOK
    

```tsx
type Props = {
  title: string;
  count: number;
};

function Counter(props: Props) {
  return <h1>{props.title}</h1>;
}
```

---

### 2. state（useState）の型定義

#### 型を書かなくていいケース

- 初期値が明確なときは **型推論に任せる**
    

```tsx
const [count, setCount] = useState(0); // number と推論される
```

#### 型を書くべきケース（重要）

- 初期値が `null` / `undefined`
    
- 後から別の型が入る予定
    

```tsx
type User = { id: number; name: string };

const [user, setUser] = useState<User | null>(null);
```

**原則**

- TypeScript は「今ある情報」からしか推論できない
    
- 初期値が曖昧なら、人間が型を教える
    

---

### 3. イベント型（最頻出）

**考え方**

> イベント型 = イベントの種類 × 要素の型

#### onChange

```ts
React.ChangeEvent<HTMLInputElement>
React.ChangeEvent<HTMLSelectElement>
```

#### onClick

```ts
React.MouseEvent<HTMLButtonElement>
```

```tsx
const handleChange = (e: React.ChangeEvent<HTMLInputElement>) => {
  setText(e.target.value);
};
```

#### 型を書かなくていいケース

- 引数を使わない
    
- handler を切り出さない
    

```tsx
<button onClick={() => doSomething()} />
```

---

### 4. APIレスポンス型

```ts
const data = await res.json();
```

- この `data` は **any / unknown**
    
- API の結果は信用しない
    

#### 最低限の安全な設計

- **state に型を持たせる**
    

```tsx
const [users, setUsers] = useState<User[]>([]);
```

**実務ルール**

> API は境界。state / props は型安全ゾーン

#### NG例

```ts
const data = (await res.json()) as User[]; // 危険
```

- 実行時エラーや仕様変更を隠してしまう
    

---

### 5. Generics（最低限）

**目的**

- 関数を汎用に保ったまま型安全にする
    

```ts
async function fetchJson<T>(url: string): Promise<T> {
  const res = await fetch(url);
  return res.json();
}

const users = await fetchJson<User[]>("/api/users");
```

- `<T>` は呼び出し側が決める
    
- `any` / `as` を減らすための道具

## フェーズ5：コンポーネント設計（初級卒業ライン）

### 1. Presentational / Container

- **Container**
    
    - state を持つ
        
    - データ取得・ロジック担当
        
- **Presentational**
    
    - props を受け取って描画するだけ
        
    - state を極力持たない
        

---

### 2. state をどこに置くか

**原則**

> 「変更する場所」と「使う場所」の共通親に置く

```txt
UserPage
 ├ SearchInput  ← 変更する
 └ UserList     ← 使う
```

→ state は `UserPage`

---

### 3. 責務分離

- 同じ意味の state を複数箇所に置かない
    
- 子は props を受け取るだけ
    

```tsx
<SearchInput value={searchText} onChange={setSearchText} />
<UserList searchText={searchText} />
```

---

### 4. props drilling

- props が多いこと自体は問題ではない
    
- **使っていない props を中継するだけ** がつらい
    

**Context を検討するライン**

- 3階層以上
    
- 中継コンポーネントが値を使っていない
    
- データの出所が分かりづらい
    

---

## 一番大事なまとめ（1行）

> **state は意味ごとに1か所だけに持ち、親が管理して子に渡す**
