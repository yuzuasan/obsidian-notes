## 1. Reactにおけるルーティングの位置づけ

* Reactは本来
  **「1つのHTML + stateによる描画切り替え」**
* ルーティングとは
  **URLを「画面状態の一部（state）」として扱うこと**

### React Routerが解決する問題

* URLと画面の対応付け
* 直接アクセス・リロード・戻る/進む
* 画面状態の再現性・共有性

> URL変更 = 描画コンポーネントの切り替え
> （ページ遷移しているわけではない）

---

## 2. React Router v6 の最小構成

### 登場要素（3つだけ）

* **BrowserRouter**

  * URL（history）を管理
  * アプリ全体を1回だけ包む
* **Routes**

  * Routeの集合（v6では必須）
* **Route**

  * `path` と `element` の対応定義

### 置き場所の原則

* `BrowserRouter` は **Appの最上位**
* ルーティングはアプリ全体の関心事

---

## 3. 画面遷移のやり方（重要な思想）

### ❌ やってはいけない

* `<a href="...">`

  * ブラウザ通常遷移
  * SPAが壊れる
  * stateが消える

### 正しい方法

#### Link（宣言的）

* メニュー・一覧など
* ユーザー操作による遷移

#### useNavigate（命令的）

* API成功後
* 条件分岐後
* ログイン完了後など

**判断基準**

* 「クリックしたら」 → Link
* 「処理の結果として」 → useNavigate

---

## 4. URLパラメータ（/users/:id）

* `:id` は **可変のURL state**
* `useParams` で取得する

### URLパラメータの本質

* URLに置かれた state
* リロードしても残る
* 共有・再現可能

👉 **「URLに置くべき state」**

### URLに置くべきでないもの

* 未保存フォーム入力
* モーダル開閉
* 一時的UI状態

---

## 5. state のスコープ意識（超重要）

Reactでは state を分類して考える：

| 種類       | 置き場所           |
| -------- | -------------- |
| 共有・再現したい | URL            |
| 画面限定     | useState       |
| アプリ全体    | Context（グローバル） |

---

## 6. グローバルstateとは

### 典型例

* ログイン中ユーザー情報
* テーマ（dark / light）
* 言語設定

特徴：

* 多くの画面・コンポーネントから参照される
* propsで渡すと破綻しやすい

---

## 7. Context API の役割と限界

### Contextが解決するもの

* props drilling
* stateの置き場所問題

### Contextが解決しないもの

* 複雑な更新ロジック
* 非同期フロー管理
* パフォーマンス最適化

> Context ≠ 状態管理ライブラリ
> **「共有の仕組み」**

---

## 8. Contextに何でも入れてはいけない理由

### ① パフォーマンス

* Contextの値変更
  → 使用している全コンポーネントが再レンダリング

### ② 設計・保守性（より重要）

* 依存関係が見えなくなる
* 修正時の影響範囲が読めない

👉 **Contextはグローバル変数ではない**

---

## 9. Router × Context の典型構成

```text
<BrowserRouter>
  <AuthProvider>
    <Routes>
      <Route ... />
    </Routes>
  </AuthProvider>
</BrowserRouter>
```

* Router：URLというグローバルstate
* Context：アプリ全体state
* Routes配下：画面

**原則**

* グローバルなものほど外側に置く

---

## 10. フェーズ7で身についた判断軸

* URLもstateであると説明できる
* stateのスコープを意識できる
* Link / useNavigate を使い分けられる
* Contextの用途と限界を説明できる
* Reduxを「とりあえず」入れなくて済む
