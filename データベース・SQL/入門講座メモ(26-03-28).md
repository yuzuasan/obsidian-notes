# 📘 データベース基礎メモ

## 🧩 リレーション設計

### ■ n:n（多対多）の関係

* **中間テーブル（中間エンティティ）を使用する**
* 例：

  * ユーザー ⇄ 商品（購入履歴など）

```text
users
products
user_products（中間テーブル）
```

### ■ 中間テーブルのポイント

* 基本は以下のカラムのみでOK

  * `id`
  * `user_id`
  * `product_id`
* 必要に応じて追加

  * 作成日時（created_at）
  * 属性（購入数など）

---

## 🔤 varcharの桁数ルール

### ■ 推奨

* **2の累乗を意識する**

  * 2, 4, 8, 16, 32, 64, 128, 255

### ■ 注意点

* **256ではなく255を使う**

  * 理由：
    → 256以上だとインデックス制約に影響する場合がある（特にMySQL）

### ■ 補足

* 実務では以下がよく使われる

  * `varchar(50)`（名前）
  * `varchar(255)`（汎用テキスト）
* 最近はDBによってはあまり厳密に気にしない場合もある

## 🔒 ロックの種類

### ■ 共有ロック（Shared Lock）

* 他ユーザーの **SELECTのみ許可**
* 更新は不可

👉 イメージ：
「読むのはOK、書くのはダメ」

---

### ■ 占有ロック（Exclusive Lock）

* 他ユーザーの **すべての操作をブロック**

  * SELECTも不可（DBによる）

👉 イメージ：
「完全に独占」

---

## 🔐 アドバイザリロック（名前付きロック）

* 任意の名前を指定してロックをかける
* アプリケーション側で制御

```sql
SELECT GET_LOCK('order_process', 10);
```

### ■ 特徴

* テーブルや行ではなく「名前」に対してロック
* 排他制御を柔軟に実装できる

---

## ⚖️ 排他制御の考え方

### ■ 楽観的ロック（Optimistic Lock）

* **更新直前にチェック**
* 衝突したらエラー

#### 方法

* versionカラムなどで管理

```sql
UPDATE users
SET name = 'A', version = version + 1
WHERE id = 1 AND version = 3;
```

👉 メリット

* ロック時間が短い
* 高速

👉 デメリット

* 衝突時のリトライ処理が必要

---

### ■ 悲観的ロック（Pessimistic Lock）

* **最初にロックしてから処理**

```sql
SELECT * FROM users WHERE id = 1 FOR UPDATE;
```

👉 メリット

* データの整合性が確実

👉 デメリット

* パフォーマンス低下
* デッドロックの可能性

---

## 🕳️ ギャップロック（Gap Lock）

### ■ 概要

* **存在しないレコードの範囲にロックをかける**

### ■ 発生例

```sql
SELECT * FROM users
WHERE age BETWEEN 20 AND 30
FOR UPDATE;
```

→ 20〜30の「隙間」もロックされる

### ■ 注意点

* 他トランザクションのINSERTを防ぐ
* ただし場合によっては

  * 他人の `SELECT FOR UPDATE` を許すケースもある

👉 主に

* **InnoDBのトランザクション分離レベル（REPEATABLE READ）**で発生

---

## 💡 追加で覚えておくと良いこと

### ■ デッドロック

* お互いにロックを取り合って進めなくなる状態

👉 対策

* ロック順序を統一
* トランザクションを短く

---

### ■ インデックスとロックの関係

* インデックスがないと

  * **広範囲にロックがかかる（テーブルロックに近い状態）**

👉 重要：

* `WHERE句`に使うカラムにはインデックスを貼る

## 📝 まとめ（超重要ポイント）

* n:nは中間テーブルで解決
* varcharは基本255まで
* ロックは「共有」と「占有」をまず理解
* 排他制御は

  * 楽観的：軽い・高速
  * 悲観的：安全・重い
* ギャップロックは「範囲ロック」と理解
