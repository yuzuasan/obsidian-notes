## 1. SQL基礎：INSERT / UPDATE / DELETE
- **INSERT**：新しいレコードを追加  
- **UPDATE**：既存レコードを変更  
- **DELETE**：既存レコードを削除
- **重要ポイント**
  - UPDATE / DELETE は **WHERE句で対象を明確に**
  - 主キーで指定できる場合は必ず主キーで指定
  - トランザクションを使うと事故を防げる  
    ```sql
    BEGIN;
    UPDATE users SET status='active' WHERE id=3;
    ROLLBACK; -- 問題あれば
    COMMIT;   -- 問題なければ
    ```
- **業務の注意**
  - 退会などは物理削除より論理削除（UPDATEでフラグ更新）が主流

---

## 2. DB設計：正規化
- **目的**：重複・不整合・更新ミスを防ぐ
- **正規化のステップ**
  1. 第1正規形（1NF）: セルに1つの値だけ  
  2. 第2正規形（2NF）: 主キーの一部に依存する列を分離  
  3. 第3正規形（3NF）: 主キー以外に依存する列を分離
- **業務注意**
  - 正規化しすぎるとテーブル数が増え、JOINが増える → パフォーマンス低下
  - 実務では「第3正規形まで」が基本

---

## 3. ER図の理解
- **目的**：テーブル同士の関係を一目で理解
- **要素**
  1. エンティティ（テーブル）
  2. 属性（カラム）
  3. リレーション（テーブル間の関係）
- **カーディナリティ**
  - 1対多（例：1顧客 → 複数注文）
  - 「多」の側に外部キーを置く
- **注意**
  - 外部キーにNULLがあると、どの顧客の注文かわからなくなる → データ不整合の原因

---

## 4. 主キー・外部キー
- **主キー（PK）**
  - テーブル内のレコードを一意に特定
  - 重複不可・NULL不可
- **外部キー（FK）**
  - 別テーブルのレコードとの関係を保証
  - 「多」の側に置く
- **削除・更新ルール**
  | ルール       | 意味 | 例 |
  |------------|------|----|
  | CASCADE    | 参照先削除/更新 → 連動 | 顧客削除 → 注文も削除（滅多に使わない） |
  | RESTRICT   | 参照されている場合は禁止 | 顧客削除 → 注文がある場合削除不可（一般的） |
  | SET NULL   | 参照先削除 → NULL | 任意の関係で使用 |
- **業務視点**
  - FK制約はデータ整合性を守る  
  - アプリ側でもチェックする場合がある

---

## 🔑 今日のまとめ
- SQLの書き換え系（INSERT/UPDATE/DELETE）は **WHERE + トランザクションが命**  
- 正規化で **重複・不整合・更新ミスを防ぐ**  
- ER図で **テーブル関係と制約** を可視化  
- 主キー・外部キーで **データ整合性** を担保  
- 実務では正規化と制約は **パフォーマンスや業務要件に合わせて調整**
