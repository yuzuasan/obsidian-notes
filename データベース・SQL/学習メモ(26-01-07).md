## 1. インデックスの基本

* **目的**：検索を高速化する「目次」のような仕組み
* **主キー・外部キー** は自動的にインデックスが張られる
* **WHEREの書き方次第で効く・効かない**

  * 関数や計算をかけると効かない（例：`DATE(created_at)`）
  * 範囲検索はインデックスの後ろに置く
* **EXPLAINで確認**

  * `type = ALL` + `key = NULL` → 全件検索・インデックス未使用
* **複合インデックス**

  * 左端から順番にしか使えない
  * = 条件（等価検索）は前、範囲検索は後
  * JOIN × WHERE でも活用可能

### 実務メモ

* JOIN条件の列 + WHERE条件の列を組み合わせて複合インデックスを作る
* 範囲検索を先頭に置かない
* テーブルの読み順もインデックス活用の鍵

## 2. RDBMSの基礎

### ① 代表的なRDBMS

| RDBMS      | 特徴          | 用途         |
| ---------- | ----------- | ---------- |
| MySQL      | 軽量・広く普及     | Webアプリ向け   |
| PostgreSQL | 標準SQL準拠・高機能 | 分析・複雑SQL向け |
| Oracle     | 商用・高信頼性     | 大規模基幹システム  |

### ② トランザクション

* **目的**：一連の処理をまとめて「全部成功 or 全部失敗」にする
* **基本操作**

```sql
BEGIN;
UPDATE ...
COMMIT;
ROLLBACK;
```

* **ACID原則**

  1. Atomicity（原子性）
  2. Consistency（一貫性）
  3. Isolation（独立性）
  4. Durability（永続性）

### ③ ロックと整合性

* **共有ロック（Sロック）**：読むとき、他も読める
* **排他ロック（Xロック）**：書くとき、他は読めない/書けない
* **デッドロック回避**

  * ロック順序を統一
  * トランザクションを短くする
  * インデックスで必要な行だけをロック

### 現場マジック

* トランザクションは「必要最小限＋短く＋順序統一」でデッドロック回避
* インデックス × トランザクション × ロックを意識すると高速＆安全

---

💡 **まとめ**

* インデックスは「作れば速い」ではなく、「使えるSQLを書いて初めて効く」
* 複合インデックスの順番 = 等価条件 → 範囲条件
* トランザクション＋ロック管理でデータ整合性を守る
* RDBMSごとの特性を理解するとSQL設計の幅が広がる
