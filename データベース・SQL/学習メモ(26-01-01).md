## 1. CASE式

* **用途**：SQL内で条件分岐したいとき
* **基本構文**

```sql
CASE
  WHEN 条件1 THEN 値1
  WHEN 条件2 THEN 値2
  ELSE 値3
END
```

* **使用例**

  * ステータスコード → 表示名に変換
  * 数値区分（ランク）
* **集計関数と組み合わせる定番**

```sql
SUM(CASE WHEN 条件 THEN 1 ELSE 0 END) AS 件数
SUM(CASE WHEN 条件 THEN 金額 ELSE 0 END) AS 金額
```

---

## 2. FROM句サブクエリ

* **役割**：加工処理を一度だけ書く
* **推奨内容**

  * CASEによる条件分岐
  * 日付・文字列加工
  * NULL置換
* **注意点**

  * 集計は外側で行う
  * 並び替えは外側で行う
* **メリット**

  * CASEの重複を防ぐ
  * 外側SQLが読みやすい
  * 修正に強い

---

## 3. LEFT JOIN × 集計の落とし穴

* **問題**

  * LEFT JOIN後にCOUNT(列)するとNULL扱いで件数がズレる
  * 複数JOINで件数が膨らむ
* **安全パターン**

```sql
SUM(CASE WHEN joined_table.id IS NOT NULL THEN 1 ELSE 0 END) AS 件数
```

* JOIN条件で絞る場合はNULLに注意
* FROM句サブクエリにCASEを書けるなら書くとさらに安全

---

## 4. パフォーマンス基礎

* **インデックスのポイント**

  1. WHEREの左辺は生の列を使う（加工禁止）
  2. 関数や計算をかけるとインデックスが効きにくい
  3. 日付範囲検索は関数ではなく範囲条件で書く
* **例**

```sql
-- インデックス効く
WHERE created_at >= '2025-01-01' AND created_at < '2025-01-02'

-- インデックス効きにくい
WHERE DATE(created_at) = '2025-01-01'
```

---

## 5. 現場あるあるバグパターン

* LEFT JOIN + COUNT → 件数ズレ
* JOIN条件で絞る → NULL行消失
* SELECTに長いCASE → GROUP BYにも同じCASE → 修正漏れでバグ

---

## 6. 安全に集計SQLを書く型（まとめ）

1. **FROM句サブクエリで加工**

   * CASE / 日付加工 / NULL補正
2. **外側で集計**

   * SUM(CASE WHEN ...) で件数や金額を集計
   * GROUP BYで分類
3. **パフォーマンス意識**

   * WHERE / JOIN の左辺は加工しない
   * 範囲条件で絞る
4. **LEFT JOIN時のNULL処理**

   * COUNTではなく SUM + CASE を使用

---

このメモをベースにすると、

* CASE / SUM + CASE / FROMサブクエリ / LEFT JOIN / インデックスの基本が整理され、
* **現場でバグりにくい集計SQL**を書く型が身につきます。
