## 🎯 フェーズ2のゴール

「コードの書き方」ではなく
**構造の歪みに気づけるレビューアになること**

* レイヤー違反を指摘できる
* SOLIDを具体的に説明できる
* テストしづらい設計を見抜ける
* トランザクション境界を業務視点で語れる

---

# ① レイヤー・依存構造レビュー

## 見るポイント

* Controller → Service → Repository の依存方向は守られているか
* 上位モジュールが下位具象に直接依存していないか
* レイヤーを飛び越えていないか
* プレゼン層とドメイン層が結合していないか

## 典型的な問題

* Service内で `System.out.println`
* ControllerがRepositoryを直接呼ぶ
* ServiceがnewでRepository生成
* EntityをそのままJSON返却

## 改善の方向性

* 抽象（interface）に依存させる
* コンストラクタインジェクション
* DTOで外部公開を分離
* UI責務はUI層へ

---

# ② アナミックドメインモデルの回避

## アナミックドメインとは

* Entityがデータだけを持つ
* 業務ロジックがServiceに散らばる

→ 手続き型設計に近い構造

## 問題点

* ロジックが分散
* 修正箇所が増える
* 不変条件が守られない
* 高凝集にならない

## 目指す形

* 状態を持つオブジェクトが振る舞いも持つ
* 不変条件はEntity内部で守る
* フィールドはprivateにする

例：

```java
public void addStock(int amount) {
    if (amount <= 0) throw ...
    if (this.quantity + amount > MAX) throw ...
    this.quantity += amount;
}
```

# ③ SOLID観点レビュー

## S（単一責任）

* 「在庫更新＋通知＋表示」など責務混在していないか
* 変更理由が複数ないか

## O（開放閉鎖）

* ルール追加のたびに既存コードを壊していないか
* if分岐地獄にならないか

## D（依存逆転）

* 具象クラスに依存していないか
* newしていないか
* 抽象に依存しているか

---

# ④ テスト容易性レビュー

## 見るポイント

* モックに差し替えられるか
* 外部APIが直書きされていないか
* 副作用が混在していないか
* メソッドが大きすぎないか

## 危険サイン

* newの多用
* staticメソッド直接呼び出し
* メール送信など副作用が同一メソッドに混在

---

# ⑤ トランザクション境界のレビュー

## 問うべきこと

* これは業務的に一連の処理か？
* 途中失敗したらどうなる？
* 整合性は守られるか？

## 重要視点

技術都合ではなく
**業務意味で境界を決める**

例：

* 通知が副次的 → 別トランザクションでも可
* 法的連携 → 同一トランザクション必須

---

# ⑥ 副作用と拡張性

## 危険パターン

```java

sendEmail();
```

## より良い設計
