# 🟢 フェーズ1：レビュー基礎体力作り（初級 → 初中級）

## 🎯 目的

* レビュー観点を体系的に持つ
* 指摘を感覚ではなく構造で行う
* 設計レベルと実装レベルを分けて考えられるようになる

---

# 1️⃣ レビューの基本構造

レビューは必ずこの形式で整理する：

```
【設計レベル】
・（重大度）指摘内容

【実装レベル】
・（重大度）指摘内容

【総評】
全体として何が一番の問題か
```

---

# 2️⃣ 設計レベルとは何か

「構造」に関する問題。

### 主な観点

* 単一責任違反（SRP）
* 層の混在（ドメイン／アプリ／インフラ）
* 抽象度の混在
* 依存方向の不整合
* トランザクション設計
* 例外の伝播設計
* ドメインロジックの置き場所
* nullを許容する設計かどうか

### 特徴

* 変更コストが高い
* 影響範囲が広い
* 将来リスクになる

---

# 3️⃣ 実装レベルとは何か

「書き方」に関する問題。

### 主な観点

* 命名は適切か
* メソッドは長すぎないか
* ネストが深すぎないか
* マジックナンバーはないか
* null安全か
* 例外処理は適切か
* ログは妥当か
* SQLインジェクション
* リソースクローズ漏れ

### 特徴

* 修正は比較的容易
* 局所的な問題

---

# 4️⃣ 重大度の付け方

レビューは必ず「強弱」をつける。

### 🔴 重大

* セキュリティ問題（SQLインジェクションなど）
* データ不整合（トランザクションなし）
* 例外握り潰し
* 業務が壊れる可能性

### 🟡 中

* 設計違反（SRP、層混在）
* 将来リスク
* 可読性低下によるバグ誘発

### 🟢 軽微

* ネスト
* マジックナンバー
* ログ改善
* 書き方の改善提案

---

# 5️⃣ フェーズ1で身につけた重要思考

### ① 可読性は品質に直結する

* ネストが深い → バグ温床
* 早期returnは有効だが乱用は危険

### ② nullは実装問題ではなく設計問題

* nullを受け取らない設計が安全
* 前提条件は契約で守る（@NotNull など）

### ③ ドメインロジックはドメイン層へ

* 金額計算
* 業務ルール（例：total > 100000）

これらはモデルが持つべき責務。

### ④ 層を壊すと構造が崩壊する

悪い例：

* 計算
* DB保存
* SQL組み立て
* ログ
* 例外処理

を1メソッドでやる

良い構造：

| 層        | 責務    |
| -------- | ----- |
| ドメイン     | 計算・判定 |
| アプリケーション | 処理の流れ |
| インフラ     | DB操作  |

---

# 6️⃣ レビュー時の思考順序（推奨）

1. セキュリティ
2. データ整合性（トランザクション）
3. 例外処理
4. 設計構造（責務・層）
5. 可読性
6. 書き方改善
