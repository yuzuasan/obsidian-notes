# フェーズ3：業務系リアル想定レビュー（中級）まとめ

## 目的

実務に近いコードレビュー能力を身につける。

特に以下を見抜けるようにする。

* 一見正しそうなコードのバグ
* 業務データの整合性問題
* パフォーマンス問題
* セキュリティ問題
* 設計の危険性

---

# レビュー観点チェックリスト（フェーズ3）

## 1. 例外・null安全

チェックポイント

* Repositoryが `null` を返す場合の考慮があるか
* `Optional` を適切に使っているか
* `NullPointerException` の可能性はないか

よくある問題

```
Product product = productRepository.findById(id);
product.getPrice();
```

問題
product が null の場合に NPE

改善例

```
productRepository.findById(id)
    .orElseThrow(() -> new ProductNotFoundException(id));
```

---

# 2. 業務データ整合性

最重要レビュー観点。

チェックポイント

* 存在しないIDを許していないか
* データ不整合を静かに無視していないか
* 異常時に例外を出す設計になっているか

危険な例

```
.map(product -> ...)
.orElse(0)
```

問題

存在しない商品を **0円として処理してしまう**

これは

```
Silent failure（静かなバグ）
```

改善

```
例外を投げて異常を早期検知
```

---

# 3. トランザクション整合性

複数DB操作がある場合に必ず確認する。

チェックポイント

* 更新処理は同一トランザクションか
* 部分成功が起きないか

危険な例

```
在庫更新
↓
注文保存
```

途中で失敗すると

```
在庫だけ減る
```

改善

```
@Transactional
```

---

# 4. N+1問題 / DBアクセス

業務アプリでは非常に頻出。

チェックポイント

* ループ内でDBアクセスしていないか

危険な例

```
for (Order order : orders) {
    productRepository.findById(...)
}
```

結果

```
注文数 × 商品数
```

DBアクセスが増える。

改善

```
ID一覧をまとめて取得
Map化する
```

---

# 5. 並行処理 / 在庫問題

EC系で特に重要。

チェックポイント

* 在庫チェックと更新が分離していないか

危険な例

```
在庫取得
在庫チェック
在庫更新
```

同時注文で

```
両方在庫OKになる
```

改善

```
DBロック
楽観ロック
UPDATE条件付き
```

---

# 6. 履歴データの再計算問題

業務システムで非常に重要。

危険な例

```
注文履歴表示
↓
現在の商品価格で再計算
```

結果

```
商品価格変更
↓
過去注文の金額が変わる
```

改善

注文時のデータを保存する。

```
OrderItem
- productId
- quantity
- priceAtOrder
```

---

# 7. セキュリティ

APIレビューでは必ず確認する。

危険な例

```
GET /orders?userId=123
```

問題

```
他ユーザーの注文が見える
```

改善

```
認証情報からユーザー取得
```

---

# 8. 計算量・パフォーマンス

Streamや綺麗なコードでも注意。

危険な例

```
items.stream()
    .map(item -> findProduct(...))
```

内部で

```
products.stream()
```

を毎回実行。

結果

```
O(n × m)
```

改善

```
Map化
キャッシュ
```

---

# 9. DI / テスト可能性

チェックポイント

* `new Repository()` していないか
* モック化できる設計か

危険な例

```
private ProductRepository repo = new ProductRepository();
```

改善

```
コンストラクタインジェクション
```

---

# 10. Optionalの誤用

Optionalは安全だが誤用も多い。

危険な例

```
Optional.orElse(0)
```

問題

```
異常状態を隠す
```

原則

```
業務異常 → 例外
```

# レビューを書くときの型

```
問題
理由
改善案
```

例

問題
商品が存在しない場合に0円として計算されている。

理由
データ不整合を静かに無視してしまい、バグ検知が遅れる可能性がある。

改善案
商品が存在しない場合は例外を投げる設計にする。
