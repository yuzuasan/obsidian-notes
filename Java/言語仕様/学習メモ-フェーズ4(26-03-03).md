# フェーズ4：並行処理・マルチスレッド まとめ

---

# 1. 並行処理の基本

## ■ Thread と Runnable

### Thread

* スレッドそのものを表すクラス
* `start()` で新しいスレッドを開始

### Runnable

* 実行処理（タスク）を表すインターフェース
* `run()` を実装する
* スレッドと処理を分離できる（推奨）

👉 実務では **Runnable を使う設計が基本**

---

# 2. synchronized とモニターロック

## ■ 排他制御とは

同時に1つのスレッドだけが実行できるようにすること。

## ■ synchronized の仕組み

* オブジェクトが持つ **モニターロック** を取得する
* インスタンスメソッド → `this` のロック
* staticメソッド → `Class` オブジェクトのロック

```java
synchronized(this) { }
```

## ■ 重要ポイント

* 別オブジェクトならロックは別
* ロック対象が違えば同時実行される

---

# 3. ロックの概念

## ■ ロックの本質

「資源への同時アクセスを防ぐ仕組み」

## ■ 問題点

* 直列化される（スループット低下）
* デッドロックの原因になる

---

# 4. volatile の意味

## ■ 解決する問題

👉 可視性（visibility）

### 保証すること

* 書き込みは必ずメインメモリへ反映
* 読み込みは必ずメインメモリから取得

### 保証しないこと

* 原子性（atomicity）
* ロストアップデートは防げない

```java
volatile int count;
```

### まとめ

| 項目  | volatile |
| --- | -------- |
| 可視性 | ○        |
| 原子性 | ×        |
| 排他  | ×        |

---

# 5. 原子性・可視性・排他の違い

| 概念  | 意味            |
| --- | ------------- |
| 原子性 | 処理が途中で割り込まれない |
| 可視性 | 他スレッドから見える    |
| 排他  | 同時実行させない      |

---

# 6. ExecutorService

## ■ なぜ必要？

Thread を直接生成すると：

* スレッド数が無制限に増える
* メモリ枯渇
* コンテキストスイッチ過多
* サーバーダウン

## ■ 解決策

スレッドプールで管理する

```java
ExecutorService executor =
    Executors.newFixedThreadPool(10);
```

### 管理するもの

* スレッド数
* タスクキュー
* ライフサイクル

---

# 7. Runnable / Callable / Future

## Runnable

* 戻り値なし
* チェック例外投げられない

## Callable

* 戻り値あり
* チェック例外OK

```java
Future<Integer> future =
    executor.submit(() -> 10);
```

## Future

* 非同期処理の結果を保持
* `get()` で結果取得（ブロックする）

### 問題

`get()` をすぐ呼ぶと非同期の意味が薄れる

---

# 8. CompletableFuture

## ■ 特徴

* 非同期処理の連結が可能
* コールバック地獄を回避

```java
CompletableFuture
  .supplyAsync(() -> 10)
  .thenApply(x -> x * 2)
  .thenAccept(System.out::println);
```

## ■ then系の違い

| メソッド       | 役割         |
| ---------- | ---------- |
| thenApply  | 値を変換して次へ渡す |
| thenAccept | 値を使うが返さない  |

---

## ■ デフォルト実行プール

`ForkJoinPool.commonPool()` が使われる。

⚠ I/Oブロッキング処理は共通プールに流すべきではない。

---

## ■ 実務設計

* CPUバウンド → 少数スレッド
* I/Oバウンド → 多めスレッド
* プールは分離する

---

# 9. デッドロック

## ■ 定義

互いに相手のロック解放を待ち続ける状態

## ■ 発生条件（4条件）

1. 排他（Mutual Exclusion）
2. 保持しながら待つ（Hold and Wait）
3. 奪取不可（No Preemption）
4. 循環待ち（Circular Wait）

## ■ 代表例

* A：lock1 → lock2
* B：lock2 → lock1

## ■ 対策

ロック取得順序を統一する

---

# 10. Thread状態

## BLOCKED

* synchronized のロック取得待ち

## WAITING

* `wait()`
* `join()`
* `park()`

### 違い

| 状態      | 何を待つ？ |
| ------- | ----- |
| BLOCKED | ロック   |
| WAITING | 通知や完了 |

---

# 11. イミュータブル設計

## ■ イミュータブルとは

状態を変更できないオブジェクト

## ■ メリット

* 競合が起きない
* synchronized 不要
* スレッドセーフ

---

## ■ 注意点

### ダメな例

```java
public List<String> getRoles() {
    return roles; // 外部から変更可能
}
```

### 正しい設計

* コンストラクタでコピー
* Getterで不変ビューを返す

```java
this.roles = List.copyOf(roles);
```

---

# 12. フェーズ4で理解した本質

* 並行処理の本当の問題は「共有状態」
* ロックは性能とトレードオフ
* volatileは万能ではない
* 非同期は設計が重要
* デッドロックは設計ミス
* イミュータブルは最強の防御
