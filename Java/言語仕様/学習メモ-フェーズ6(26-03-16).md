# JVM・メモリ管理の理解（フェーズ6）

## JVMとは

**JVM（Java Virtual Machine）**
Javaプログラムを実行する仮想マシン。

Javaは直接ネイティブコードにコンパイルされず、**バイトコード**を経由して実行される。

### Javaの実行フロー

```
Javaソースコード (.java)
    ↓ javac
バイトコード (.class)
    ↓
JVM
    ↓
OS上で実行
```

特徴

* JVMがOSごとに存在する
* 同じバイトコードを異なる環境で実行できる

これにより

```
Write Once, Run Anywhere
```

が実現される。

---

# クラスローダー

クラスローダーは **.classファイルをJVMに読み込む仕組み**。

```
.class
↓
JVMにロード
↓
Classオブジェクト生成
```

## クラスローダーの種類

JVMには3階層のクラスローダーがある。

```
Bootstrap ClassLoader
      ↑
Platform ClassLoader
      ↑
Application ClassLoader
```

### Bootstrap ClassLoader

JDKのコアクラスをロードする

例

```
java.lang.*
java.util.*
java.io.*
```

### Platform ClassLoader

JDKの拡張ライブラリをロードする

### Application ClassLoader

アプリケーションのクラスをロードする

例

```
com.example.*
```

---

# 親委譲モデル（Parent Delegation Model）

クラスをロードする際は

```
子 → 親
```

の順で問い合わせる。

```
Application
   ↓
Platform
   ↓
Bootstrap
```

目的

* 標準ライブラリの保護
* セキュリティ確保
* クラスの一意性保証

もし親委譲がなければ

```
java.lang.String
```

などをアプリ側で上書きできてしまう。

---

# JVMメモリ構造

JVMの主なメモリ領域

```
Heap
Stack
```

## Heap

オブジェクトを保存する領域。

```
User user = new User();
```

メモリ構造

```
Stack
└ user（参照）

Heap
└ Userオブジェクト
```

特徴

* JVM全体で共有
* GCの対象

---

## Stack

メソッド実行時の作業領域。

保存されるもの

* ローカル変数
* メソッド引数
* オブジェクト参照
* 中間計算結果

特徴

```
メソッド終了
↓
Stackフレーム破棄
```

---

# GC（Garbage Collection）

Javaは不要なオブジェクトを自動で回収する。

回収条件

```
GC Root から到達できないオブジェクト
```

これを

```
到達可能性解析
(Reachability Analysis)
```

という。

---

## GC Root

GCの起点となる参照。

代表例

```
Stackのローカル変数
staticフィールド
CPUレジスタ
JNI参照
```

---

# 世代別GC（Generational GC）

JavaのHeapは世代ごとに管理される。

```
Heap
 ├ Young Generation
 └ Old Generation
```

さらにYoungは分かれる

```
Young
 ├ Eden
 ├ Survivor0
 └ Survivor1
```

---

## 世代別仮説

```
ほとんどのオブジェクトはすぐ死ぬ
```

この仮説に基づきGCが最適化されている。

---

# オブジェクトのライフサイクル

```
new
↓
Eden
↓ (GC)
Survivor
↓
Old Generation
```

短命オブジェクト

```
Edenで回収
```

長寿命オブジェクト

```
Oldへ昇格
```

---

# GCの種類

## Young GC

対象

```
Young Generation
```

特徴

* 頻繁に発生
* 比較的高速

---

## Old GC

対象

```
Old Generation
```

特徴

* 発生頻度は低い
* 処理が重い

---

## Full GC

対象

```
Heap全体
```

特徴

```
Stop The World
```

発生中は

```
全アプリケーションスレッド停止
```

---

# Stop The World（STW）

GC実行中

```
アプリケーションスレッド停止
```

影響

```
レスポンス遅延
タイムアウト
スループット低下
```

そのため

```
Full GCはできるだけ避ける
```

---

# メモリリーク

Javaでもメモリリークは発生する。

原因

```
不要なオブジェクトへの参照が残る
```

例

```
staticコレクション
```

```
static List<User> cache = new ArrayList<>();
```

参照構造

```
GC Root
 ↓
static
 ↓
List
 ↓
User
```

参照が残るため

```
GCされない
```

結果

```
Heap圧迫
↓
OutOfMemoryError
```

---

# メモリリークと設計ミスの違い

### メモリリーク

```
不要なのに参照が残る
```

例

```
Listener解除忘れ
```

### 設計問題

```
キャッシュが無制限
```

例

```
static List
```

# JITコンパイル

JIT（Just In Time Compiler）は

```
バイトコード
↓
ネイティブコード
```

へ変換する最適化機構。

---

## JVM実行の実際

```
バイトコード
↓
インタプリタ実行
↓
頻繁に実行されるコード検出
↓
JITコンパイル
↓
ネイティブコード実行
```

頻繁に実行されるコードを

```
HotSpot
```

と呼ぶ。

例：**HotSpot JVM**

---

# JVMまとめ

Javaの性能とメモリ管理は以下で支えられている。

```
JVM
 ├ クラスローダー
 ├ メモリ管理（Heap / Stack）
 ├ GC
 └ JITコンパイル
```

これらにより

```
ポータビリティ
メモリ自動管理
高性能実行
```

が実現されている。
