# フェーズ3：ラムダ・Stream API・関数型の理解

---

## 1. ラムダ式の内部仕様

### ■ 正体

* 無名クラスの糖衣構文ではない
* `invokedynamic` を使って実行時に生成される
* 毎回クラスファイルが増えるわけではない

### ■ メリット

* クラス数の増加を防ぐ
* 起動遅延を抑える
* JVM が最適化しやすい設計

---

## 2. 関数型インターフェース

### ■ 定義

* 抽象メソッドを1つだけ持つインターフェース
* `default` / `static` メソッドはカウントしない
* `Object` 由来のメソッドもカウントしない

---

### ■ 代表例（java.util.function）

| インターフェース        | 役割          |
| --------------- | ----------- |
| `Predicate<T>`  | T → boolean |
| `Function<T,R>` | T → R       |
| `Supplier<T>`   | () → T      |
| `Consumer<T>`   | T → void    |

---

## 3. Stream API の本質

### ■ パイプライン構造

```
stream()
   ↓
中間操作（filter/map など）
   ↓
終端操作（forEach/collect など）
```

### ■ 重要ポイント

* 中間操作は即実行されない（遅延評価）
* 終端操作が呼ばれたときに初めて処理が走る

### ■ なぜ遅延評価？

* パフォーマンス最適化
* 短絡評価（limit / findFirst など）
* 並列処理対応

---

## 4. collect の内部動作

`collect()` は3要素で構成される。

| 役割      | 関数          |
| ------- | ----------- |
| 入れ物を作る  | supplier    |
| 要素を追加   | accumulator |
| 並列結果を結合 | combiner    |

### ■ なぜ combiner が必要？

→ 並列ストリームで分割結果をマージするため

---

## 5. 並列ストリームの注意点

### ■ 仕組み

* ForkJoinPool の commonPool を使う

### ■ 危険ポイント

* 共有可変状態（例：複数スレッドで同じ ArrayList を変更）
* I/O処理中心
* データ量が少ない
* 順序保証（forEachOrdered）
* commonPool 枯渇による全体停止

### ■ 原則

* 状態を共有しない
* collect を使う
* 順序不要なら `parallel().forEach(...)`

---

## 6. Optional の正しい使い方

### ■ 目的

* null可能性を型として明示する
* NullPointerException を防ぎやすくする

---

### ■ 良い使い方

```
name.map(String::toUpperCase)
    .ifPresent(System.out::println);
```

---

### ■ orElse と orElseGet の違い

| メソッド        | 評価タイミング |
| ----------- | ------- |
| orElse()    | 即時評価    |
| orElseGet() | 遅延評価    |

重い処理を書く場合は `orElseGet()` を使う。

---

### ■ フィールドに Optional を使わない理由

* 状態が3つに増える（値あり / 空 / null）
* 設計が壊れる
* シリアライズ非対応

→ Optional は **戻り値専用**

---

# フェーズ3の核心理解

* ラムダはJVM最適化前提設計
* Streamは遅延評価による全体最適化モデル
* collectは並列前提の設計
* 並列は「速くなるかも」機能
* Optionalはnull撲滅ではなく「null明示化」
