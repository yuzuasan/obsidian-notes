# Phase 2：構造に関するパターン（Structural）

## 目的
- 既存コードを壊さず拡張する力を身につける
- キーワード：**「呼び出し側を汚さない」**

---

## 対象パターンと本質

### 1. Adapter
- **目的**：インターフェースを合わせる（翻訳機）
- **効果**：
  - レガシーや外部APIの呼び出しを呼び出し側に漏らさない
  - 変更点を局所化
  - DIP（依存関係逆転原則）を守れる
- **実務例**：`LegacyPaymentSystem` の typeCode 変換
- **覚え方**：既存コードを変更せずに接続

### 2. Decorator
- **目的**：インターフェースを維持したまま機能を追加
- **効果**：
  - 横断的処理（ログ、認可、トランザクションなど）を追加
  - 呼び出し側は元のインターフェースだけを見る
- **実務例**：`PaymentService` にログや承認チェックを後付け
- **覚え方**：変更はしない、機能は足す
- **重要ポイント**：包むクラスは元オブジェクトと同じインターフェースを実装

### 3. Facade
- **目的**：複雑なサブシステムを呼び出し側に簡単な窓口で提供
- **効果**：
  - Controller → Service → Domain の複雑な依存を隠す
  - 呼び出し側がシンプルになり心理的ハードルが下がる
- **実務例**：Controller から Domain の複数クラスに直接アクセスする代わりに Facade を経由
- **覚え方**：呼び出し側の心理的ハードルを下げる窓口

### 4. Composite
- **目的**：単体と集合を同じ操作で扱える
- **効果**：
  - 木構造の再帰的操作が可能
  - 呼び出し側は単体か複合かを意識せず扱える
- **実務例**：
  - 組織ツリー（部署→部署→メンバー）
  - 帳票セクション（セクション→項目）
  - フォルダ構造（フォルダ→サブフォルダ→ファイル）
- **覚え方**：個と集合を同じように扱える

### 5. Proxy
- **目的**：元オブジェクトを直接変更せずに横断的処理を挟む
- **効果**：
  - アクセス制御、キャッシュ、遅延評価などを追加
  - 呼び出し側は元オブジェクトと同じインターフェースで扱える
- **実務例**：高額決済承認チェック、キャッシュ、Lazy Load
- **覚え方**：元オブジェクトを包んで機能を挟む

---

## 実務テーマとの対応

- **既存システムとの接続（レガシー／外部API）**  
  → Adapter を使って呼び出し側に漏らさない設計
- **呼び出し側を汚さない設計**  
  → Adapter, Decorator, Facade, Proxy で呼び出し側の変更最小化
- **フレームワークが勝手に使うパターン理解**  
  → Decorator（AOP / Spring Proxy）、Facade（Service層統合）、Proxy（遅延ロード・認可）

---

## Phase 2 学びのまとめ（要点）
- **共通思想**：既存コードを壊さずに扱う  
- **Adapter** = 接続・翻訳  
- **Decorator** = 機能追加  
- **Facade** = 呼び出し側の単純化  
- **Composite** = 単体・集合を同じ操作  
- **Proxy** = 元オブジェクトを変更せず横断的処理
