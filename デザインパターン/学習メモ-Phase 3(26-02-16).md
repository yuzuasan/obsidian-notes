# Phase 3：振る舞いに関するパターン（Behavioral・前半）

## 🎯 学習テーマ

* if / switch が増え続ける理由
* 「業務ルール」をクラスとして扱う
* 処理フローの柔軟な切り替え

---

# 1️⃣ なぜ if / switch は増え続けるのか？

## 本質的な原因

* 将来増える「変動点」を直接分岐で書いている
* アルゴリズム・状態・処理順序などを1箇所に集中させている
* 拡張のたびに既存コードを修正する必要がある（OCP違反）

### 典型例

```java
if (type == CREDIT) { ... }
else if (type == PAYPAY) { ... }
else if (type == BANK) { ... }
```

決済が増えるたびに if が増える。

---

## 💡 解決方針

> 変動点をクラスとして分離する

---

# 2️⃣ 各パターンの「変動点」

| パターン                    | 変動点               |
| ----------------------- | ----------------- |
| Strategy                | アルゴリズム（振る舞い）の差し替え |
| Template Method         | 処理手順（骨格）の一部ステップ   |
| Chain of Responsibility | 処理担当の連鎖・責任の流れ     |
| Command                 | 実行される処理そのもの       |

---

# 3️⃣ Strategy パターン

## 本質

> アルゴリズムを差し替え可能にする

## 使う場面

* 決済種別
* 計算ロジックの違い
* フォーマット変換

## 例

```java
interface PaymentStrategy {
    void pay();
}
```

```java
Map<PaymentType, PaymentStrategy> strategies;
strategies.get(type).pay();
```

## メリット

* if/switch削減
* テストしやすい
* モック差し替えが容易
* 新規追加時に既存コード変更不要

---

# 4️⃣ Template Method パターン

## 本質

> 処理の「順序」は固定し、一部ステップだけ差し替える

## 使う場面

* バッチ処理
* データ取込フロー
* 共通手順を守らせたい場合

## イメージ

```java
abstract class AbstractProcess {

    public final void execute() {
        step1();
        step2();
        step3();
    }

    protected abstract void step2();
}
```

## 特徴

* 処理順は親クラスが握る（安全）
* 実装を強制できる
* フローの破壊を防げる

---

# 5️⃣ Chain of Responsibility

## 本質

> 処理を連鎖させる

## 使う場面

* バリデーション
* 承認フロー
* フィルター処理

## イメージ

```java
validator1.setNext(validator2);
validator2.setNext(validator3);
```

```java
validator1.validate(request);
```

## 特徴

* if を並べる代わりにクラスを並べる
* 条件分岐の肥大化を防ぐ
* 新しい処理はクラス追加のみ

---

# 6️⃣ Command パターン

## 本質

> 「やること」をオブジェクトとして扱う

## 構造

* Invoker（実行者）
* Command（execute）
* Receiver（実処理）

## 例

```java
interface Command {
    void execute();
}
```

## 強み

* 遅延実行
* キュー投入
* 再実行
* Undo
* ログ保存
* 実行主体の分離

## 変動点

> 実行される処理そのもの

---

# 7️⃣ パターン選択の判断基準

## アルゴリズムが変わる → Strategy

## 状態で振る舞いが変わる → State（後半）

## 処理順を守らせたい → Template Method

## 処理を順番に渡したい → Chain

## 実行を抽象化したい → Command

---

# 8️⃣ 設計の本質

### ❌ 悪い方向

* if を10個並べる
* 将来増える前提の分岐を書く
* 1クラスが全部判断する

### ✅ 良い方向

* 変動点を見つける
* 業務ルールをクラス化する
* 呼び出し側をシンプルにする
* 拡張時に既存コードを触らない設計

---

# 9️⃣ 今回の到達レベル

* if肥大化の原因を説明できる
* Strategy / Template / Chain / Command の違いを言語化できる
