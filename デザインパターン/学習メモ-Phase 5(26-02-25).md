# Phase 5：GoFパターンを「現場設計」に統合する

## 🎯 このフェーズの目的

* パターンを“知っている”状態から
* パターンを“自然に使える”状態へ移行する

ゴール：

> 「この変更、どのパターンで守る？」が自然に出ること

---

# 1️⃣ パターンの組み合わせ方

## ケース：承認ワークフロー付き申請システム

### 使用した構成

* **State**

  * 状態ごとの振る舞いルールを閉じる
  * 不正遷移防止
  * OCPを守る

* **Domain Event**

  * 「承認された」という出来事を表現

* **Observer**

  * 通知・ログ・外部連携を副作用として分離

* トランザクション分離

  * 承認は確実にコミット
  * 副作用は別責務

### 学んだこと

* パターンは単体で使わない
* コア（業務）を守る構造を作る
* 副作用は外に逃がす

---

# 2️⃣ 過剰設計 vs 必要設計

### 問い

* イベント永続化は常に必要か？
* 再試行は本当に必要か？
* 通知失敗は業務影響があるか？

### 判断基準

> 業務的に問題があるかどうか

通知が失敗しても業務が成立するなら：

* イベント永続化は不要

ここで重要なのは：

> 高度な設計 = 良い設計 ではない

# 3️⃣ フレームワーク × GoF

## Spring Boot / .NET の裏側

### DIコンテナ

* Abstract Factory
* Singleton

### `@Transactional` / AOP

* Proxy

### イベント機構

* Observer

### 本質

> フレームワークは GoF パターンの集合体

理解すると：

* なぜ self-invocation で Transaction が効かないか説明できる
* なぜ循環参照でDIが壊れるか理解できる

---

# 4️⃣ パターンを使わない勇気

重要な気づき：

* イベント永続化は常に必要ではない
* 非同期基盤は常に必要ではない
* State も常に必要ではない

判断基準：

> その複雑さは業務価値を守るためか？
