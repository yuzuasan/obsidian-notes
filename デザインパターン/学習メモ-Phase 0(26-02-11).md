# Phase 0：デザインパターンを学ぶための前提整理

## 目的

* デザインパターンを**暗記ではなく道具として使う準備**
* 「なんかこのクラスやばい」を**言語化できる状態**になる

---

## デザインパターンとは何か／何ではないか

### デザインパターンとは

* **変更に弱くなりがちな設計を避けるための知恵・型**
* 業務で繰り返し発生する「壊れ方」への対処法

### デザインパターンではない

* if を消す魔法
* きれいなクラス構成を作るためのテンプレ
* 設計が破綻しているコードを救済する万能薬

👉 **前提となる設計が壊れていると、どのパターンも効かない**

---

## なぜ業務系でデザインパターンが必要になるのか

### 業務コードの特徴

* 種別・区分・状態が必ず存在する
* 仕様変更・追加が前提
* if / switch で分岐しがち

### よくある問題

* Service クラスに条件分岐が集約される
* 変更のたびに同じクラスを修正する
* 影響範囲が読めず、修正が怖くなる

👉 **業務は「分ける」必要があるが、分け方を誤ると壊れる**

---

## 悪い業務コードから見えた本質的な問題

表面的な問題：

* 条件分岐が多い

本質的な問題：

* **変更理由が異なるロジックが1クラスに混在している**
* その結果、変更のたびに同じクラス（例：BillingService）が壊される

👉 問題は if の数ではなく
👉 **「どこが変更され続けるか」**

---

## 「変更に強い設計」とは何か

### 定義（業務系視点）

> **変更内容と修正箇所が1対1で対応している設計**

例：

* VIP の料金ルール変更 → VIP に関するコードだけ修正
* 種別追加 → 既存コードは触らず追加で対応

### よくある誤解

* if が少ない＝変更に強い ❌
* クラスが多い＝変更に強い ❌

👉 **影響範囲が直感的に分かることが重要**

---

## SOLID原則との関係性（Phase 0 で押さえる範囲）

### SRP（単一責任の原則）

* 1クラスは **1つの理由でのみ変更されるべき**
* 種別ごとのルールを1クラスに集めない

### OCP（開放閉鎖原則）

* 既存コードを修正せず、**追加で対応できる設計**
* 種別追加のたびに Service を修正する設計は OCP 違反

👉 SOLID は
👉 **「この設計、将来どこが壊れるか？」を見抜くための視点**

---

## Phase 0 の結論

* デザインパターンは **SOLID が守られている前提で使う道具**
* パターン適用前に
  **「設計がそもそも破綻していないか」を疑う**
* 業務コードの違和感を
  **感覚ではなく言葉で説明できることがゴール**
