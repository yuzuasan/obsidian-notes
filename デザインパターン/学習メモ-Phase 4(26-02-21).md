# Phase 4：振る舞いに関するパターン（Behavioral・後半）

## 🎯 学習テーマ

* 状態遷移をコードで安全に表現する
* イベント駆動の考え方を理解する
* ドメイン知識をコード構造に反映する
* 複雑性の源泉（状態・時間・並行性）を理解する

---

# 1. 状態をどう扱うか

## ■ 問題の本質

多くのバグはここから生まれる：

```
状態 × 時間 × 並行性
```

* 状態が分散する
* if分岐が増える
* 変更時に既存コードを修正する必要がある
* 二重実行・不整合が起きる

---

## ■ Stateパターン

👉 「状態による振る舞いの違いを閉じ込める」

### ポイント

* 状態と振る舞いを一体にする
* 分岐をポリモーフィズムへ置き換える
* OCPを守れる
* 状態ロジックの分散を防ぐ

### 効果

* 状態変更時の影響範囲が局所化
* ドメイン知識が構造として表現される

---

# 2. 横連携の複雑化

## ■ 典型的な問題

受注確定時に：

* メール送信
* 在庫更新
* ログ記録
* 外部連携
* ポイント付与

これを毎回ベタ書きすると：

* 既存コード修正が必要
* 副作用が肥大化
* 影響範囲が読めなくなる

---

## ■ Observerパターン

👉 「変化を自動通知する」

### ポイント

* 通知元がイベントを発行
* 通知先は疎結合
* 横方向に拡張できる

### 効果

* 副作用を分離できる
* OCPを守れる
* 機能追加が安全

### 本質

* 主導権はイベント発行側
* 横に“広がる”構造

---

# 3. 網の目依存の解消

## ■ Mediatorパターン

👉 「中央で調停する」

### ポイント

* オブジェクト同士を直接結ばない
* 依存を中央に集約する
* 網の目依存を放射状に変える

### 注意点

* 神クラス化リスク
* 条件分岐の集中

### 使いどころ

* 横連携ではなく「相互調整」が必要な場合

---

# 4. 状態退避と時間

## ■ Mementoパターン

👉 「状態を壊さず退避する」

### ポイント

* カプセル化を守る
* 外部から内部状態を直接触らせない
* スナップショット保存

### イベントソーシングとの違い

| Memento | イベントログ    |
| ------- | --------- |
| ある瞬間を保存 | すべての変化を保存 |
| 状態保存    | 事実保存      |

---

# 5. 複雑性の源泉

## 状態 × 時間 × 並行性

### 状態

→ 分散すると破綻

### 時間

→ 上書きすると履歴が消える

### 並行性（最難関）

→ 再現不能
→ 順序非決定
→ レース発生

---

# 6. 並行性への対抗手段

## 原則

❌ 並行性を賢く扱う
⭕ 並行性を減らす

---

## 戦略

* キューで直列化
* 排他制御
* バージョン管理
* 集約単位でロック
* 冪等設計

---

## 冪等性

👉 「二回実行されても壊れない」

分散環境では：

* 再送
* タイムアウト
* 重複処理

は避けられない。

だから：

整合性は「結果」
冪等性は「戦略」

---

# 7. パターンの役割整理

| パターン     | 役割              |
| -------- | --------------- |
| State    | 状態ごとの振る舞いを閉じ込める |
| Observer | 横方向に疎結合で広げる     |
| Mediator | 中央で調停する         |
| Memento  | 状態を安全に退避する      |

---

に到達している。

* 並行性を減らす
